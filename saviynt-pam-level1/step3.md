# Step 3: Mitigation via Saviynt Cloud PAM Principles

Now, let's secure the target system using the core capabilities of **Saviynt Cloud PAM** via our simulated CLI agent.

### 1. Onboard and Lock Down the Host
Run the onboarding command. This registers the server and configuration settings to the central governance controller:
`saviynt-cli onboard`{{exec}}

Try to manually append another un-tracked backdoor key string to the file now:
`echo "hacker_key" >> /home/deploy-admin/.ssh/authorized_keys`{{exec}}
*You are blocked with a "Permission denied" error because Saviynt has locked down local authorization modifications. Only the agent can mediate access.*

### 2. Request a 15-Second JIT Access Session
Request a dynamic, short-lived Just-in-Time key authorized by an administrator (`ciso@saviynt.com`):
`saviynt-cli request-jit ciso@saviynt.com 15`{{exec}}

### 3. Immediately Test the Connection
While the 15-second countdown is active, execute the provided SSH command to verify the agent successfully opened the access path:
`ssh -o StrictHostKeyChecking=no -i /tmp/jit_temp_key deploy-admin@localhost`{{exec}}
*(Type `exit` and hit Enter to disconnect once you verify you are logged in).*

### 4. Wait for Expiration and Re-Test
Wait a few seconds for the timer to run out, then attempt to log in again using the exact same command:
`ssh -o StrictHostKeyChecking=no -i /tmp/jit_temp_key deploy-admin@localhost`{{exec}}
*The connection is instantly rejected. The Saviynt engine cleanly stripped the ephemeral public key the exact second the authorization token expired.*

### 5. Inspect the Central Audit Trail
View the secure, structured compliance and governance logs generated by Saviynt during this lifecycle:
`saviynt-cli audit-logs`{{exec}}
